## 🎯 一、 为什么要这样设计项目？(架构篇)

### 1. 分层架构（为什么不把代码全塞进一张纸里？）
如果你把发微信、收短信、记账的任务全交给一个人，他肯定会崩溃。代码也是一样。我们的项目被拆分成了几个专门的“打工人”：

*   **领域层 (`Note`)**：最底层的纯数据对象，它就像一张便利贴，只负责保存写在上面的“文字（Title/Content/Tags/Date）”。它**坚决不碰**读写文件或者找卡片这种麻烦事。
*   **数据层 (`Repository`)**：仓库管理员，像一个有无数抽屉的大柜子（本质是 `std::vector`），专门负责把所有的 `Note` 收纳在一起，提供增加、删除、查找等服务。
*   **持久层 (`Storage` - 暂未实现)**：档案科员工，负责在程序关闭时把大柜子里的卡片打包存入硬盘文件，并在程序重新启动时读回内存。

### 2. 为什么要使用时间戳可以恢复的设计？
每次新建卡片都会获取当前时间 `std::time(nullptr)`。但仔细想：当你明天打开程序，从硬盘文件里读出你昨天写的卡片时，如果不允许程序修改卡片的时间，那么这卡片就会变成“今天刚刚写”的！
因此，`Note` 类必须提供一个修改时间的后门接口（比如 `setDate`），专门在**从文件加载数据时**，把文件中老旧的时间原封不动地“贴”回卡片上（恢复原状）。

## 🔧 二、 核心 C++ 语法避坑指南 (代码篇)

你在写下 `Note.hpp` 和 `repository.hpp` 的实战中，跨越了这几个初学者最容易掉坑的知识点：

### 1. `std::vector` 的优势（为什么标签用它？）
标签（Tags）是一个数量可变的字符串列表。在 C++ 里表达列表最简单直白且性能极高的方式就是 `vector`（动态数组）。它不光能随时扩容，还默认保持了你传入时的先后顺序。

### 2. `const` 和引用的深度理解（权限控制）
在 C++ 里，传来传去很容易把数据复制成很多份（既慢又占内存），或者不小心被别人改乱了。
*   **不复制只借用 (`&`)**：`const std::vector<std::string>& getTags() const;`
    *   通过 `&`，你把原本的标签列表**直接借给了**调用者看，而不去复印一份。
*   **只读防篡改 (`const`)**：前面的 `const` 表示“你只能看，不能用我的借给你的东西加/删标签”。
*   **只读防内鬼 (函数末尾的 `const`)**：末尾修饰符告诉编译器：“我保证我这个查标签的方法，绝对不会去偷偷修改 `Note` 卡片里的任何私有资料。”
*   **指针的只读返回 (`std::vector<const Note*> result`)**：在 `findByTag` 里返回这个结构，就像图书管理员给你递来了“只能看不能画的保护书”。你拿着这些卡片可以去打印书名，但绝不能去越权篡改它们的内容。这是极其安全的设计。

### 3. 指针调用 `->` 与对象调用 `.` (该戳谁？)
如果一个变量名叫 `note`：
*   **如果是对象本体**（如 `Note note;`）：用**点号（`.`）** -> `note.getTags()`
*   **如果是它的指针或地址**（如 `Note* note;` 或智能指针）：用**箭头（`->`）** -> `note->getTags()`

### 4. 寻值利器 `std::find`
这是一个超高频的算法工具。
*   **用法**：`std::find(tags.begin(), tags.end(), "C++")`。意思是你拿着 `"C++"` 这个目标，去 `tags` 列表的开头查到结尾。
*   **判定方式**：只要它返回的结果**不等于**列表的底线 `.end()`，就说明它半路截住目标了（找到了）。
*   **局限**：它一次只能找**一个明确的值**。如果你想找多个要求“必须全部出现”的复杂情况，得结合外部循环自己包一层。

### 5. 抛出异常 (`throw`)
在按位置查找卡片 (`getNoteByIndex`) 时，如果有人要找第 100 张卡片，但其实只有 2 张，怎么办？
*   最好的做法是直接发脾气打断：`throw std::out_of_range("Index out of range");`。向调用者抛出一个“范围越界”的异常，这比返回个乱码或者崩溃要安全明确得多。

### 6. 永远的尺子 `size_t`
*   这是 C++ 专门用来表示“**大小、长度、第几个索引**”的无符号整数（永远不可能是负数）。
*   **为什么这是最佳实践？**
    *   防呆：永远不可能出现索引是 `-1` 这种荒唐事。
    *   匹配容器：和 `vector.size()` 的天然返回类型一致，消除编译器对于“有符号整数和无符号整数比较”的警告。
    *   **禁忌**：永远不要用它做 `>= 0` 的倒数循环，因为减到 0 再减 1 时，它会下溢变成一个无穷大的正数，导致死循环。

## 🎮 三、 Git 生存手册：把写代码变成“玩游戏存盘”

你在终端和 VS 面板里经历了一场激烈的 Git 搏斗，这其实是一个极其标准的云端存档流程。以后只要记住这三个“游戏行为”的比喻，就不会再迷茫：

### 1. 选出要存档的物品（Git Add / 暂存）
> **“决定把我背包里的哪些道具带入下一次存盘。”**
当你在文件里敲了新代码，文件旁会出现一个小红勾（或 `M` 标记），这意味着文件被修改了。
*   **怎么做：** 在 VS 的“Git 更改”面板里，点击文件旁边的 `+` 号（暂存）。

### 2. 写存档日记并保存在电脑硬盘（Git Commit / 提交）
> **“按下单机游戏的存档键，并给存档起名叫：打完第一关 Boss。”**
这是在你的这台电脑硬盘上打了一个历史快照。这时候拔了网线，代码的历史记录也在你的电脑里。
*   **怎么做：** 在面板那个输入框里，写下一句总结（比如：“完成了 Repository 的标签查找”），然后点 **`全部提交 (Commit All)`**。

### 3. 把本地存档同步给网盘服务器（Git Push / 推送）
> **“把刚才存在电脑上的通关存档，上传备份到官方服务器（GitHub）。”**
*   **怎么做：** 点击界面上方那个**向上的箭头（`↑`）**。
*   **结果**：等它转完圈圈，进度就永远锁死在网上了，电脑坏了也不怕。

### 🚨 避雷针：为什么你刚开始合并代码失败了？
你之前遇到了一个经典的新手大坑：在 GitHub 创建仓库时，云端自动产生了一条包含空 README 的 `main`（主宇宙分支）；而你本地一开始名叫 `master`（你电脑里的另一个宇宙分支）。
这两条毫无血缘关系的线无法自动融合，导致拉取请求（Pull Request）失败。

**我们是如何解决的？统一天下！**
1.  在本地把叫 `master` 的分支强行改名为 `main`。
2.  用**强行覆盖（`git push -f`）**的方式，把我们本地这个扎实的 `main`（装着刚写的代码）像重锤一样砸到 GitHub 上，把网上那个没用的空分支直接替代掉。
3.  **从此以后**，你只有唯一一个时间线，直接本地写完 -> Commit -> 向上箭头 Push 即可。
